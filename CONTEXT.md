# CONTEXT.md — Party Climb (working title)

เกม party platformer 2D สไตล์ 8-bit สำหรับ PC — ผู้เล่นแข่งกันปีนขึ้นที่สูง มีไอเทมป่วนกันระหว่างทาง

---

## Race Mode
โหมดแข่งความเร็ว — ผู้เล่นคนแรกที่แตะยอดของด่านเป็นผู้ชนะ HP ทำหน้าที่เป็น penalty ไม่ใช่เงื่อนไขแพ้ถาวร

## Speed Run Mode
เหมือน Race Mode แต่ไม่จบรอบทันทีที่มีคนถึงเส้นชัยคนแรก — ทุกคนเล่นต่อจนกว่าจะถึงเส้นชัยครบทุกคน หรือหมดเวลา (180 วินาที) แล้วจึงโชว์ leaderboard เรียงตามเวลาที่แต่ละคนถึงเส้นชัย คนที่ยังไม่ถึงตอนหมดเวลาขึ้นเป็น DNF ตามลำดับความสูง ณ ขณะนั้น ผู้เล่นที่ถึงเส้นชัยแล้วหยุดขยับ (freeze) ยืนอยู่บนเส้นชัยให้ทุกคนเห็น แต่ HP/ไอเทมยังทำงานปกติ (เผื่อโดนกระเด็นทีหลัง — ดู Open Questions)

## Survival Mode
โหมดเอาชีวิตรอด — ผู้เล่นคนสุดท้ายที่ยังมีชีวิตอยู่เป็นผู้ชนะ HP หมด = ออกจากรอบทันที กลายเป็นผู้ชม (เห็นแค่จอหลักที่กล้อง follow คนที่เหลือ ทำอะไรไม่ได้จนจบรอบ) เส้นชัยของด่านไม่มีผลในโหมดนี้ Survival เปล่า (ไม่มี lava) ใช้แรงกดดันจาก PvP + ตกด่านล้วน ๆ — ดู Lava Map สำหรับเวอร์ชันที่มี Rising Hazard

## Lava Map
Survival Mode ที่เพิ่ม Rising Hazard (ลาวา) เข้ามาเป็นแรงกดดันหลักแทน/เสริม PvP — เงื่อนไขชนะเหมือน Survival Mode ทุกอย่าง (รอดคนสุดท้าย) ลาวาเริ่มไล่ขึ้นหลัง countdown จบ 6 วินาที ด้วยความเร็วคงที่ (ปรับจูนได้ที่ `LAVA_RISE_SPEED`/`LAVA_START_DELAY` ใน `src/game/constants.ts`) กล้อง (จอที่ทุกคนแชร์ร่วมกัน) ถูกลาวาลากขึ้นตรง ๆ ด้วย ไม่ใช่แค่ follow ผู้เล่นที่นำอยู่เหมือนโหมดอื่น — นี่คือความหมายของ "ด่านขยับขึ้นเอง" การแตะลาวาไม่ใช่ instant kill (ดู Rising Hazard ด้านล่าง)

## Rising Hazard
อันตราย (เช่น ลาวา) ที่ไต่ขึ้นจากด้านล่างของด่านอย่างต่อเนื่องใน Lava Map บังคับให้ผู้เล่นต้องปีนขึ้นตลอดเวลา ความเร็วของ Rising Hazard เป็นตัวควบคุม pacing หลักของโหมด การแตะ Rising Hazard ไม่ใช่ instant kill — ผู้เล่นเสีย HP, ถูก knockback ขึ้นตรง ๆ, และได้ invulnerability ช่วงสั้น HP เป็นทรัพยากรกลางที่ถูกกัดกินจากทุกแหล่ง (ตก, โดนโจมตี, กับดัก, hazard) — HP หมดขณะอยู่ใน Lava Map = ตกรอบเหมือน Survival Mode

## HP
พลังชีวิตของผู้เล่น แสดงเป็นหัวใจ เริ่มรอบละ 3 ดวง — ทุกแหล่งความเสียหาย (ตกด่าน, โดนยิง, เหยียบกับดัก, แตะ Rising Hazard) กัด 1 ดวงเท่ากัน ความแตกต่างของแต่ละแหล่งอยู่ที่ side effect (knockback, stun) ไม่ใช่ตัวเลข ความหมายของ "HP หมด" ต่างกันตามโหมด (ดู Race Mode / Survival Mode)

## Shared Camera
ผู้เล่นทุกคนใช้จอเดียวร่วมกัน (แบบเกม Contra) กล้อง follow ผู้เล่นที่อยู่สูงสุดของฉาก ขอบล่างของจอคือเส้นอันตราย

## Player Interaction
ตัวละครผู้เล่นไม่ชนกันทางกายภาพ (ทะลุกัน) — ปฏิสัมพันธ์ PvP ทั้งหมดเกิดผ่านไอเทมเท่านั้น ดูเหตุผลใน ADR-0001

## ตกด่าน (Fall Out)
การหลุดขอบล่างของจอ = เสีย HP ใน Race Mode แรงกดดันนี้มาจากผู้เล่นที่นำอยู่ (ผู้นำลากกล้องขึ้น) ส่วนใน Survival Mode แรงกดดันมาจาก Rising Hazard — ทั้งสองโหมดใช้ camera logic เดียวกัน

## Mystery Box
วิธีเดียวที่ผู้เล่นได้ไอเทม — กล่องสุ่มที่ spawn ตามด่าน เก็บแล้วสุ่มได้ไอเทม 1 ชิ้น การสุ่มถ่วงน้ำหนักตามอันดับ (rubber-banding): ผู้เล่นที่รั้งท้ายมีโอกาสได้ไอเทม comeback แรงกว่า ผู้นำมีโอกาสได้ไอเทมรับมากกว่า ผู้เล่นถือไอเทมได้ครั้งละ 1 ชิ้น ใช้ด้วยปุ่มเดียว และไอเทมที่ถือแสดงเป็นไอคอนเหนือหัวตัวละคร มองเห็นได้โดยทุกคน

## ไอเทม: สลับที่ (Swap)
สลับตำแหน่งกับผู้เล่นที่อยู่สูงสุด ณ ขณะใช้เสมอ (Blue Shell ของเกมนี้) — rubber-banding ต้องจำกัดให้เฉพาะผู้เล่นท้ายตารางมีโอกาสสุ่มได้ เพื่อกันการสลับวนระหว่างอันดับ 1–2

## ไอเทม: โจมตี (Shot)
Projectile ยิงตรงไปทิศที่ตัวละครหันหน้า ต้องเล็งเอง — ไอเทมเดียวในเกมที่มี skill expression โดนแล้วเสีย HP + knockback (knockback สามารถทำให้เหยื่อหลุดแพลตฟอร์มจนตกด่านได้ = kill ทางอ้อม) ฝั่งผู้ยิงตรวจ hit บนจอตัวเอง เหยื่อเป็นคน apply ผล (ตาม ADR-0001) hitbox ออกแบบให้ใจดีเพื่อชดเชย latency

## ไอเทม: เกราะ (Shield)
กดใช้แล้วเกิดฟองเกราะรอบตัว (ทุกคนมองเห็น) กันได้ 1 hit แล้วแตก — กันได้ทุกอย่างรวมถึงสลับที่ เป็น counterplay หลักของผู้นำ เหยื่อเช็คสถานะเกราะบนเครื่องตัวเอง (victim-authoritative ตาม ADR-0001) จึงไม่มีปัญหา timing dispute จาก latency

## ไอเทม: กับดัก (Trap)
วางบนแพลตฟอร์ม ทุกคนเห็นตอนวาง แล้วค่อยๆ จางหายใน ~2 วินาที (เกมความจำ — ใช้จุดแข็งของ Shared Camera) เหยียบแล้วเสีย HP + stun สั้น ~1 วิ วางได้คนละ 1 อันในสนาม กับดักที่หลุดขอบล่างจอถูกลบทิ้งอัตโนมัติ

## Chunk
ท่อนด่านที่ออกแบบมือ ขนาดประมาณ 1 จอ — ด่านจริงเกิดจากการสุ่มลำดับ chunk ต่อกันแนวตั้งด้วย seed ที่แชร์ร่วมกันตอนเริ่มรอบ ทุก client generate ด่านตรงกันเป๊ะโดยไม่ต้องส่งข้อมูลด่านผ่าน network จุด spawn ของ Mystery Box ฝังอยู่ใน chunk (ดู ADR-0002)

## Room
ห้องเล่นรองรับผู้เล่น 2–8 คน — จำนวนสูง (6–8 คน) สร้างแรงตึงกับ design อื่น ที่ต้อง playtest เป็นพิเศษ: ความถี่ของสลับที่ใส่ผู้นำ, ความแออัดของ Shared Camera, และความอ่านง่ายของ HUD (ดู Open Questions)

## Platform
Prototype แรกทำเป็น web game เล่น online ด้วยกันได้ เป้าหมายระยะยาวคือ PC

# Open Questions

- ~~**MVP ordering:**~~ resolved — ทำ Race ก่อน ตามด้วย Speed Run / Survival / Lava Map
- **Survival/Lava tie เมื่อคนสุดท้าย 2 คนตกรอบพร้อมกัน:** ปัจจุบันไม่มี client ไหน observe เงื่อนไข "เหลือฉันคนเดียว" เป็นจริง จึงไม่มีผู้ชนะประกาศ (รอบค้าง) → ต้อง resolve ก่อน ship จริง (แนวทางที่เป็นไปได้: ประกาศเสมอเมื่อ alive เหลือ 0)
- **Speed Run กับไอเทมหลังเข้าเส้นชัย:** ผู้เล่นที่ finish แล้วยัง freeze อยู่กับที่แต่ยังโดนยิง/สลับที่/กับดักได้ (เวลาที่บันทึกไปแล้วไม่เปลี่ยน แต่ physics อาจดูแปลกถ้าโดนกระเด็น) — ยังไม่ resolve ว่าควร immune หลัง finish ไหม
- **Lava rise speed/start delay:** ตั้งไว้กะคร่าว ๆ (`LAVA_RISE_SPEED=8px/s`, `LAVA_START_DELAY=6s`) → ต้องจูนตอน playtest เหมือนตัวเลขอื่น ๆ
- **8 คนกับสลับที่:** ที่ 6–8 คน ผู้นำอาจโดนสลับถี่เกินไป — อาจต้องมี global cooldown ของสลับที่ หรือลดน้ำหนักสุ่มเมื่อคนเยอะ → รอ playtest
- **8 คนกับ Shared Camera:** คนหลุดขอบล่างจอพร้อมกันหลายคน — อาจต้อง zoom out ตามจำนวนคน → รอ playtest
- **Rubber-banding curve:** น้ำหนักสุ่มไอเทมตามอันดับยังไม่กำหนดตัวเลข → จูนตอน playtest
- **Respawn ใน Race Mode:** HP หมดใน Race แล้ว "ตาย" หมายถึงอะไรกันแน่ (respawn ที่ไหน เสียอะไรบ้าง) — ยังไม่ resolve
- **Controls บน web:** keyboard-only หรือรองรับ gamepad ด้วย — ยังไม่ resolve
- **Round structure:** เล่นรอบเดียวจบ หรือสะสมคะแนนหลายรอบแบบ Mario Party — ยังไม่ resolve
